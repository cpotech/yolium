# Windows CRLF Scheduled-Agent Fix

## What was implemented

- Fixed scheduled-agent markdown parsing so the renderer accepts Windows `CRLF` line endings when loading, validating, and parsing specialist definitions.
- Normalized scaffolded and updated specialist markdown to `LF` output so generated files stay stable across Linux and Windows checkouts.
- Added regression coverage for Windows-style newline handling in both the renderer markdown helpers and the main-process scaffold service.

## Files modified and why

- `src/renderer/components/schedule/AddSpecialistDialog.tsx`: normalize markdown newlines before regex parsing for integrations, frontmatter validation, name extraction, and guided-form parsing.
- `src/main/services/specialist-scaffold.ts`: normalize template output and rewritten specialist content to `LF` line endings.
- `src/tests/AddSpecialistDialog.test.ts`: added CRLF utility regression tests that reproduce the Windows failure.
- `src/tests/specialist-scaffold.test.ts`: added coverage that template output stays normalized to `LF`.

## Tests added and results

- Added 2 CRLF parsing tests in `src/tests/AddSpecialistDialog.test.ts`.
- Added 1 LF normalization test in `src/tests/specialist-scaffold.test.ts`.
- `npm test`

## Caveats / follow-up

- The full unit suite still emits pre-existing React `act(...)` warnings in several UI tests; they do not fail the run and were not part of this fix.
- The likely root cause of the referenced GitHub Actions failure was Windows checkout/newline handling in the newly added scheduled-agent markdown workflow. The new CRLF regression tests now cover that path directly.
