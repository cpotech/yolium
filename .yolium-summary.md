# Implementation Summary: Switch Project Shortcuts to Tab Selection

## What Was Implemented
Repurposed numbered keyboard shortcuts (1-9, 0) in the kanban content zone from selecting sidebar projects by index to selecting open tabs by index. When a tab is selected via shortcut, it becomes active and its content is focused.

## Files Modified

### `src/renderer/components/kanban/KanbanView.tsx`
- Removed `sidebarProjects?: SidebarProject[]` and `onProjectSelect?: (path: string) => void` props
- Added `tabs?: Tab[]` and `onTabSelect?: (tabId: string) => void` props
- Imported `Tab` type from `@shared/types/tabs`
- Removed unused `SidebarProject` import
- Updated numbered shortcut handler (lines 521-533): checks `tabs` array instead of `sidebarProjects`, calls `onTabSelect(tabs[tabIndex].id)` instead of `onProjectSelect(path)`
- Updated `handleKeyDown` dependency array

### `src/shared/vim-actions.ts`
- Updated descriptions for `kanban-project-1` through `kanban-project-10` (lines 66-75)
- Changed "Select project N" to "Select tab N"

### `src/renderer/App.tsx`
- In KanbanView rendering (around line 626), replaced `sidebarProjects={sidebarProjects}` and `onProjectSelect={handleProjectClick}` with `tabs={tabs}` and `onTabSelect={setActiveTab}`

### `src/tests/e2e/tests/tab-numbered-shortcuts.spec.ts`
- Added 3 missing E2E tests to the existing test file:
  - `should select second tab when pressing 2 in content zone` — creates a second tab and verifies pressing 2 activates it
  - `should focus tab content after selection via numbered shortcut` — verifies kanban view receives focus after tab selection
  - (Existing tests covered: first tab selection, out-of-bounds inertness, dialog-open inertness, input-focused inertness)
