## What was implemented

Implemented the Claude usage spinner fix by separating the IPC snapshot from the renderer lifecycle state and making loading explicit. The renderer now distinguishes between `loading`, `ready`, `unavailable`, and `no-oauth`, so a completed `{ hasOAuth: true, usage: null }` response no longer renders an endless spinner.

## Files modified and why

- `src/shared/types/agent.ts`: added `ClaudeUsageSnapshot` plus explicit Claude usage lifecycle statuses for renderer state.
- `src/preload.ts`: updated the preload typing so `window.electronAPI.usage.getClaude()` returns the transport snapshot instead of the renderer state.
- `src/renderer/hooks/useClaudeUsage.ts`: mapped snapshots into explicit states, kept the initial load spinner separate from completed null responses, and preserved the last successful usage payload across refresh failures.
- `src/renderer/components/StatusBar.tsx`: replaced the nested Claude ternary with explicit rendering branches for loading, ready, unavailable, and no-oauth.
- `src/tests/status-bar-claude-usage.test.tsx`: rewrote the status bar regression tests around explicit statuses and the unavailable fallback behavior.
- `src/tests/useClaudeUsage.test.tsx`: added dedicated hook lifecycle coverage for loading, ready, unavailable, no-oauth, refresh failure retention, and visibility-triggered refetching.
